# Axiia Cup 智能体大赛 — 设计规范（唯一标准文档）

> 最后更新：2026-03-30
> 状态：MVP 阶段，面向内部 15 人测试
> 本文档为唯一标准。历史讨论文档仅供参考，以本文为准。

---

## 1. 产品概述

**产品名称：** Axiia Cup（AA Cup）

**一句话：** 选手编写系统提示词，打造 AI 智能体，在人文学科场景中进行对抗性对话比赛。

**目标用户：** 中国初高中及大学生

**MVP 目标：** 端到端展示完整用户体验。前端必须展示完整流程，后端可以不完善。

---

## 2. 用户流程

```
注册（手机号 + 微信） → 选择场景 → 编写提示词（两个角色各一份）
→ 选择模型 → 试炼场测试 → 提交 → 异步对战 → 查看排行榜
```

## 3. 场景系统

每个 **学科**（历史、法律、戏剧、谈判等）下包含一个或多个 **场景**。

### 场景定义

| 要素 | 说明 |
|------|------|
| 学科 | 历史、法律、戏剧、销售、哲学等 |
| 背景 | 背景材料、情境设定、约束条件 |
| 角色 A | 名称、公开目标、隐藏目标（可选） |
| 角色 B | 名称、公开目标、隐藏目标（可选） |
| 边界约束 | 禁止行为（如穿越时空、跳出角色） |
| 对话轮数 | 场景作者自定义（10-20 轮，默认 20） |
| 裁判人设 | 场景内的角色（如女王、皇帝），负责赛后提问评分 |
| 裁判问题 | 对话结束后向双方提出的问题，及评分规则 |

- **MVP 范围：** 至少 1 个场景（咳嗽/Simon 设计）
- 胜负条件、裁判人设、裁判问题均由场景作者设计

## 4. 智能体工坊

| 项目 | 规则 |
|------|------|
| 输入 | 每个角色一份系统提示词（纯文本） |
| 字数限制 | 1000 字符（中文） |
| 工具调用 | 禁止。无 RAG、无多轮链、无函数调用 |
| 模型选择 | 选手从平台提供的国产模型中选择（Kimi、DeepSeek、通义千问、MiniMax） |
| 海外模型 | 禁止（无 Claude、GPT、Gemini） |
| 参数调优 | 禁止。温度、top-k 等由平台锁定（temperature = 0） |
| 提交方式 | 必须同时提交角色 A 和角色 B 的提示词 |
| 版本管理 | 可提交多个版本；比赛 block 开始时自动锁定最新版本 |

**模型选择是比赛策略的一部分。** 不同模型有不同特点，选对模型是竞争力。

## 5. 试炼场

试炼场供选手在正式提交前测试提示词。

### MVP 功能

- 自己的角色 A 提示词 vs 自己的角色 B 提示词（自己打自己）
- 查看完整对话记录
- 查看裁判评分预览

### 单人练习模式（待定）

- 可能方案：玩家 vs 固定 NPC（如商鞅 Game 1 模式）
- 可提供独立排行榜（所有人面对同一个 boss）
- 是否实现由场景作者决定，非平台层面决策

### 待定问题

- 试炼场测试次数是否有限额？
- 试炼场是否使用与正式比赛相同的模型 API？

## 6. 裁判系统

### 核心设计（已确认 2026-03-30）

**裁判 = 场景角色人设 + 公开提示词 + 统计收敛**

| 设计原则 | 说明 |
|----------|------|
| **裁判 prompt 公开** | 所有选手可查看裁判 prompt。对称博弈下，双方都能优化，不破坏公平 |
| **裁判有角色人设** | 不是抽象的评分机器，而是场景中的角色（如哥伦布场景的伊莎贝拉女王、商鞅场景的秦孝公） |
| **赛后提问评分** | 对话结束后，裁判角色向双方各提若干问题，根据关键回答评分 |
| **统计稳定性** | 单场 LLM 判定有方差，但大量对局后统计收敛。德州扑克模型：规则公开 + 大量对局 = 长期公平 |
| **版本迭代** | 裁判 prompt 版本化管理，根据玩家反馈公开更新，类似竞技游戏平衡补丁 |

### 裁判流程

```
对话阶段（N 轮）
    ↓
裁判角色向 Agent A 提问（若干问题）
    ↓
裁判角色向 Agent B 提问（若干问题）
    ↓
根据关键回答计分 → 判定胜负
```

### 系统偏差应对

LLM 裁判可能有系统性偏差（如偏好更长回复、偏好某种修辞风格）。应对方式：
- 公开 prompt → 玩家发现偏差并反馈
- 运营方更新裁判 prompt → 发布更新说明
- 这就是竞技游戏的标准平衡流程

## 7. 对战引擎

| 项目 | 规则 |
|------|------|
| 执行方式 | 全异步。对战在后台运行，选手后续查看结果 |
| 对话轮数 | 场景作者自定义（10-20 轮，默认 20） |
| Temperature | 0（确定性输出） |
| API 失败处理 | 从失败点重试（不重启整场对战） |
| 对手 prompt 可见性 | 不可见 |
| 配对方式 | 每对选手打两场：A 的角色A vs B 的角色B，以及 A 的角色B vs B 的角色A |

## 8. 赛制

### 格式：多轮瑞士制 + Elo Rating

不是一次性跑完所有比赛，而是分多个 **block**，每个 block 之间留出 prompt 迭代窗口。

### 赛程结构（15 人示例）

```
赛前阶段
├── 注册 + 提交 prompt
├── 试炼场自测
└── 管理员确认参赛名单

比赛阶段（~2 周）
├── Block 1（Day 1）：瑞士轮 4 轮
├── 迭代窗口（3-4 天）：查看记录，改 prompt
├── Block 2（Day 5）：瑞士轮 4 轮
├── 迭代窗口（3-4 天）
├── Block 3（Day 9）：最终 block
└── 公布最终排名
```

### 瑞士轮规则

- 每轮：战绩相近的选手配对（赢的打赢的，输的打输的）
- 不重复：同一 block 内不重复配对；跨 block 允许（prompt 已迭代）
- 轮空：奇数人时 1 人轮空（算赢）
- 不淘汰：所有人打完全部轮次

### 数字

| 参数 | 值 |
|------|-----|
| 参赛人数 | 15 |
| Block 数 | 3 |
| 每 block 瑞士轮数 | 4 |
| 每 block 对战组数 | ~28 |
| 每组比赛场次 | 2（正反角色） |
| 总场次 | ~168 |
| 每人总对局 | 12 |
| 迭代机会 | 2 次 |

### 排名系统

- **Elo/Glicko-2 rating**：跨 block 累计，实时更新
- 排行榜：Elo 分数 + 趋势箭头（↑↓→）
- Buchholz 作为辅助小分（对手平均强度）

### 管理员操作

- 设定比赛开始时间（触发 Block 1）
- 设定迭代窗口长度
- 比赛开始后不接受新玩家

## 9. 技术架构（MVP）

> MVP 为演示用架构。工程团队（Anna/米正）将在 MVP 后重新设计生产系统。

| 层 | MVP 方案 |
|------|------|
| 前端 | Web 应用——完整 UX 流程 |
| 后端 API | 最小化：用户 CRUD、prompt 提交、对战结果 |
| 对战引擎 | 异步任务：配对 → 对话 → 裁判评分 → 存储 |
| 模型网关 | 国产模型 API 适配器（Kimi、DeepSeek、通义千问、MiniMax） |
| 裁判 | 公开人设化 LLM 裁判；每场景版本化 prompt |
| 数据库 | 用户、提交（带版本）、对战记录、结果 |
| 试炼场 | 前端可用；后端可 stub |

### 基础设施（Fly.io）

- VM：512MB RAM，shared CPU
- 数据库：SQLite，启动时自动 seed（空库时）
- 依赖：openai SDK（替代 litellm，减少内存占用）

## 11. 竞品差异

**vs 狼人杀 AI 大赛：**
- 更简单（1v1 而非多人），token 成本低
- 强调人类设计者的能力，而非模型本身
- 人文学科方向，教育市场价值高

**vs OpenClaw / 工作流 Agent 大赛：**
- 它们测「agent 能做什么」（工具调用、自动化）
- 我们测「你能多好地设计对抗性对话策略」
- 完全不同的能力维度

## 13. 设计风格

- **方向：** 暗色电竞仪表盘（Clean Dashboard）
- **字体：** Satoshi（英文/标题）+ 思源黑体（中文）+ Geist Mono（数据）
- **强调色：** 朱砂红 #E04A2F
- **背景：** #0C0C0C（纯暗底）
- **密度：** 紧凑，信息优先
- **动效：** 最小化，功能性为主
